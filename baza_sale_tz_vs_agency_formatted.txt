Сравнение нового ТЗ и текущего фронтенд-модуля

Тип документаСравнительный аналитический документ по фронтендуИсточникИсходный markdown-текст пользователя

Что уже есть в прототипе Agency (фронтенд), чего нет, и примерно какой объём доработок по интерфейсу.

Важно: здесь про фронтенд (экраны, кнопки, роли). Бэкенд, БД и интеграции не оцениваем.



Условные обозначения по объёму

• Маленький объём: до нескольких дней работы одного фронтенд‑разработчика.
• Средний объём: от ~1 до 2 недель работы одного фронтенд‑разработчика.
• Большой объём: 3+ недель работы одного фронтенд‑разработчика (или меньше, если работает команда).

Это очень грубо, без учёта багфиксов, ревью и изменения требований.



1. Каркас, меню, верхняя панель

• Что уже есть
  - Главный дашборд с плитками и графиками.
  - Левое меню с основными разделами.
  - Верхняя панель с базовыми элементами.
• Что добавляет ТЗ
  - Чёткая структура меню: Маркетплейс, CRM (классическая/модная), отдельные отчёты по лидам, по команде, Золотой фонд, Отложенный спрос и т.д.
  - Верхняя панель: глобальный поиск по всем сущностям, кнопка «быстрое создание», центр уведомлений, возможный переключатель роли/контуров.
• Объём
  - Меню: маленький–средний (перестроить структуру, добавить подпункты и переходы).
  - Верхняя панель: средний (нарисовать UI глобального поиска, быстрого создания, центра уведомлений; логика пока может быть моковой).



2. Лиды

• Что уже есть
  - Сильный модуль лидов: канбан‑воронка, карточка лида, источники, аналитика, регистрация покупателя, вкладки по менеджерам, партнёрам, настройкам.
• Что добавляет ТЗ
  - Жёстко зафиксированные статусы лида (new / in_progress / qualified / no_answer / lost / postponed).
  - SLA по первому контакту, напоминания и сигналы руководителю при просрочке.
  - Отдельные «режимы» работы: работа с источниками лидов, работа с распределением лидов, аналитика лидов (как подстраницы/табы).
• Объём
  - Доработки внутри уже существующего модуля: маленький–средний.



3. Клиенты

• Что уже есть
  - Данные клиента внутри лида (форма регистрации покупателя).
  - Но отдельного раздела «Клиенты» нет.
• Что добавляет ТЗ
  - Отдельный модуль «Клиенты»:
    - список клиентов, фильтры, сегменты (Золотой фонд, Отложенный спрос);
    - карточка клиента с историей, интересами, подборками, встречами, сделками, задачами.
  - Чёткий процесс перевода из лида в клиента.
• Объём
  - Новый раздел и карточка + связи с уже существующими сущностями: средний–большой.



4. Объекты и Маркетплейс

• Что уже есть
  - Сильный модуль «Мои объекты»: таблица, карточки, фильтры, статусы, мастер создания/редактирования.
• Что добавляет ТЗ
  - Разделение на:
    - «Управление объектами» (внутренний каталог для закупки);
    - «Маркетплейс» (витрина для менеджеров и партнёров).
  - Строгая структура по первичке/вторичке, новостройки, юниты.
  - Подразделы: Управление первичкой, Первичка на продажу, Мои объекты, Подборки, Сравнения.
• Объём
  - Переделать существующий модуль и добавить «витринный» режим: средний–большой.



5. Подборки

• Что уже есть
  - Логика подбора объектов под клиента частично встроена в лиды и объекты.
  - Отдельного модуля «Подборки» нет.
• Что добавляет ТЗ
  - Сущность «подборка»: список подборок, карточка подборки (объекты, комментарии, статус, отправка клиенту).
  - Связь с клиентом и сделкой, отслеживание отклика.
• Объём
  - Новый модуль + интеграция с лидами/клиентами/объектами: средний.



6. Сделки

• Что уже есть
  - Сделки фигурируют в аналитике и KPI как «успешные лиды» и суммы.
  - Отдельного раздела «Сделки» нет.
• Что добавляет ТЗ
  - Полноценный модуль сделок:
    - список сделок;
    - карточка сделки: клиент, объект, статусы, этапы, бронирования, платежи, юридические задачи, документы, история.
  - Отчёты по сделкам и по комиссиям.
• Объём
  - Новый крупный модуль + несколько страниц/отчётов: большой.



7. Брони и регистрации

• Что уже есть
  - Прямых модулей по броням/регистрациям нет.
• Что добавляет ТЗ
  - Раздел «Брони / регистрации»:
    - забронировать клиента;
    - забронировать квартиру;
    - список броней, статусы, сроки действия, связь со сделкой.
• Объём
  - Отдельный модуль с 2–3 экранами: средний–большой.



8. Задачи

• Что уже есть
  - Поля hasTask и taskOverdue у лидов, виджет задач на главном дашборде.
  - Локальная логика задач внутри карточки лида.
• Что добавляет ТЗ
  - Отдельный модуль задач:
    - список задач по пользователю/команде;
    - статусы, дедлайны, повторяющиеся задачи;
    - привязка задач к любым сущностям (лид, клиент, сделка, объект и т.д.).
• Объём
  - Отдельный центр задач + интеграция с существующими экранами: средний.



9. Календарь

• Что уже есть
  - График‑теплокарта активности (аналитика), но не настоящий календарь встреч.
• Что добавляет ТЗ
  - Полноценный календарь:
    - встречи, показы, юридические события;
    - привязка к лидам/клиентам/сделкам;
    - личный и командный вид.
• Объём
  - Новый визуальный компонент + связи: средний–большой.



10. Аналитика и дашборды и рабочие места ролей

• Что уже есть
  - Сильный модуль аналитики: дашборды собственника/сети, графики по лидам, партнёрам, объектам.
  - Главный дашборд менеджера/руководителя в упрощённом виде.
• Что добавляет ТЗ
  - Чёткие ролевые дашборды и отдельные «рабочие места» для ролей:
• Собственник
• CEO / генеральный директор
• Руководитель отдела продаж
• Менеджер (риэлтор)
• Руководитель отдела закупки
• Юрист
• Маркетолог
• Тестовый кабинет партнёра (readonly)
  - Для каждой роли в ТЗ прописаны свои виджеты и экраны (что видеть, какие действия делать, какие метрики смотреть).
  - Отдельные отчёты: по лидам, по команде, по сделкам, по комиссиям, по объектам, по партнёрам.
• Объём
  - Переиспользование существующих графиков + новые экраны/комбинации:
    - ролевые дашборды: средний–большой;
    - отдельные отчётные страницы: средний.



11. Команда, роли и доступы

• Что уже есть
  - Страница персонала, оргструктура, карточки сотрудников, блокировка.
  - Детализированная матрица ролей и прав на фронтенде, отключение кнопок по правам.
• Что добавляет ТЗ
  - Более формализованная матрица прав (формула «действие + сущность + уровень доступа»).
  - Отдельная зона «Управление доступами» в интерфейсе.
• Объём
  - В основном доведение интерфейса до нужной формы: маленький–средний.



12. Партнёры и города

• Что уже есть
  - Страницы города, партнёров, аналитика сети, управление ролями.
• Что добавляет ТЗ
  - Явное разделение по типам партнёров: рефералы, посредники, собственники.
  - Учёт вклада партнёров в лидах/сделках, простые кабинеты партнёров.
• Объём
  - Расширить существующие экраны + добавить новые списки/карточки: средний.



13. Кабинеты партнёров (включая тестовый кабинет)

• Что уже есть
  - Аналитика по партнёрам и городам, но нет отдельного «кабинета» партнёра.
• Что добавляет ТЗ
  - Тестовый кабинет партнёра (readonly): ограниченный дашборд, ограниченный маркетплейс, новости.
• Объём
  - Новый упрощённый layout/набор экранов: маленький–средний.



14. Обучение / база знаний

• Что уже есть
  - Модуль LMS: страница с материалами, простая админка.
• Что добавляет ТЗ
  - Чёткая структура: курсы, уроки, статьи, регламенты, тесты.
  - Видимость материалов по ролям.
• Объём
  - Расширить существующий модуль и добавить несколько типов контента: маленький–средний.



15. Информация, новости, уведомления

• Что уже есть
  - Конструктор рассылок, городские рассылки, настройки уведомлений.
• Что добавляет ТЗ
  - Раздел «Информация»: новости, уведомления, напоминания, управление новостями.
  - Центр уведомлений в верхней панели (уведомления о событиях, просрочках, дедлайнах).
• Объём
  - Лента новостей + простой центр уведомлений: средний.



16. Настройки и тарифы

• Что уже есть
  - Страница настроек: профиль, компания, уведомления, биллинг, безопасность (всё на моках).
• Что добавляет ТЗ
  - Тарифы как отдельный подраздел с управлением логикой тарифов (на уровне UI).
  - Настройки напоминаний, тем, уведомлений в нужной структуре.
• Объём
  - В основном доработка существующих вкладок: маленький–средний.



17. Итог простыми словами

• Текущий фронтенд‑прототип уже закрывает значительную часть картинок и логики по: лидам, объектам, аналитике, ролям, персоналу, обучению, рассылкам и настройкам.
• Крупные новые куски интерфейса по ТЗ:
  - отдельные модули «Клиенты», «Сделки», «Брони / регистрации», «Подборки», «Задачи», «Календарь»;
  - ролевые дашборды для всех ролей;
  - полноценный маркетплейс, кабинеты партнёров, центр уведомлений.
• По объёму это тянет на крупный фронтенд‑проект: если грубо, одна команда из 2–3 фронтенд‑разработчиков будет занята этим на несколько месяцев, при условии, что требования не будут сильно меняться и бэкенд/дизайн двигаются параллельно.

19. Что важно и обязательно учесть

Ниже пункты, которые в ТЗ зафиксированы жёстко и их нельзя потерять в реализации:

• 8 ролей и отдельные рабочие места
  Собственник, CEO, Руководитель продаж, Менеджер, Руководитель закупки, Юрист, Маркетолог, Тестовый кабинет партнёра.

• Юрист — не совсем роль в списке
  В ТЗ есть отдельный сценарий: при этапе сделки «задаток» автоматически создаётся задача юристу «Подготовить договор задатка», после этого запускаются дополнительные задачи БП сопровождения.

• MVP из ТЗ включает больше, чем просто лиды и объекты
  Обязательно: клиенты, подборки, сделки, брони по первичке, задачи, календарь, базовые ролевые дашборды, партнёрский контур, юридическая автозадача, обучение, уведомления/напоминания.

• Ролевая модель по формуле
  Права строятся как действие + сущность + уровень доступа с уровнями own/assigned/team/branch/project/all, а не только «роль = набор кнопок».

• В ТЗ есть явные цепочки процессов
  лид → клиент → подборка → встреча → сделка и сделка → бронь → задаток → юридические задачи → завершение.  
  Это важно: интерфейс должен поддерживать именно эти переходы.

• Есть обязательные разделы интерфейса и подменю
  Например: CRM «Классическая/Модная», Клиенты («Золотой фонд», «Отложенный спрос»), Сделки (отчёт по сделкам/комиссиям), Задачи (дашборд задач + управление), Информация (уведомления/напоминания/новости/управление новостями).

• Нефункциональные требования тоже значимы даже для фронта
  Поиск/фильтры/пагинация на списках, быстрые рабочие экраны, ролевое скрытие чувствительных данных, история изменений критичных сущностей.
